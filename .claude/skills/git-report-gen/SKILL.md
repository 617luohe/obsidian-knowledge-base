# Git Report Generator — 从 Git 历史到工作汇报 PPT 全流程

## 概述

本 skill 封装了从 git 提交历史到工作汇报 PPT 的完整流程，将数月的代码提交记录自动转化为结构化的工作汇报材料。

## 六阶段流程

```
① 提取 git 历史  ──→  ② 生成日报  ──→  ③ 汇总周报
                                        ↓
⑥ 生成 PPTX      ←──  ⑤ PPT 大纲    ←──  ④ 撰写汇报文档
```

## 阶段详解

### 阶段 ①：提取 Git 提交历史

**输入**：一个或多个 git 仓库地址（或本地路径）、时间范围（起止日期）

**操作**：
1. Clone 或定位目标仓库
2. 使用 `git log --since --until --format` 提取所有提交记录
3. 合并多个仓库的提交，按日期排序
4. 统计提交数、活跃日期数、时间跨度

```bash
git log --since="2025-07-01" --until="2026-07-02" \
  --format="%H|%ai|%s" > all_commits.txt
```

### 阶段 ②：生成日报

**输入**：提交记录数据、工作日历（含中国节假日）

**操作**：
1. 将提交按日期分组
2. 对每个工作日：
   - 有提交：从 commit message 提取工作内容，按类别标注（算法排程/规则优化/评分体系/数据工程等）
   - 无提交：根据前后提交内容推理填充，节假日跳过
3. 写入 Obsidian 格式的日报 markdown 到 `5-Journal/daily/{year}/{year}-{month}/`

**关键规则**：
- 使用 `scripts/generate_reports.py` 执行
- 工作日判断需考虑中国法定节假日（春节、国庆、五一、清明、端午等）和调休
- 无提交日的推理逻辑：查找最近的前后有提交日，延续其工作主题

### 阶段 ③：汇总周报

**输入**：已生成的日报数据

**操作**：
1. 按 ISO 周号分组
2. 统计每周工作天数、提交数、涉及模块
3. 筛选出代表性提交（feat/fix/perf/refactor）
4. 生成周总结 markdown 到 `5-Journal/weekly/{year}/`

### 阶段 ④：撰写工作汇报文档

**输入**：周报汇总数据 + 日报原始数据

**操作**：
1. 分析整个周期的工作主线，划分为 2-4 个阶段
2. 每个阶段提炼核心主题与关键产出
3. 从 commit message 中提取关键里程碑
4. 识别个人成长维度（端到端交付、算法设计、质量意识等）
5. 规划下阶段工作（P0/P1/P2 优先级）
6. 输出结构化的 markdown 汇报文档

**输出结构**：
```markdown
# 周期工作汇报
## 一、工作概述（整体定位 + 关键数据）
## 二、工作进展（分阶段详述）
## 三、个人成长总结（N个能力维度）
## 四、下阶段工作计划（优先级表格 + 节奏预期）
```

### 阶段 ⑤：设计 PPT 大纲

**输入**：工作汇报 markdown 文档

**操作**：
1. 根据汇报文档内容规划幻灯片结构（通常 10-14 页）
2. 确定每页的布局模式（卡片、时间轴、双列、表格等）
3. 选择配色方案和视觉主题
4. 输出 PPT 大纲（每页标题 + 关键内容要点 + 布局建议）

**大纲格式**：
```
| 页码 | 标题 | 布局 | 关键内容 |
|------|------|------|----------|
| 1 | 封面 | 深色背景+大字标题 | 汇报标题/人名/周期 |
| 2 | 目录 | 编号导航 | 3-4项带图标的导航 |
| ... | ... | ... | ... |
```

### 阶段 ⑥：生成 PPTX

**输入**：PPT 大纲 + 汇报内容数据

**技术栈**：
- `pptxgenjs`（Node.js）— 生成 .pptx 文件
- `react-icons` + `sharp` — SVG 图标转 PNG base64
- `python-pptx` — QA 检查

**操作**：
1. 使用 `scripts/generate_pptx.js` 作为模板参考
2. 根据大纲编写具体的幻灯片代码
3. 运行脚本生成 .pptx
4. 使用 `python-pptx` 进行程序化 QA（布局/字体/重叠检查）
5. 如有 LibreOffice，使用 `markitdown` 提取文本做内容 QA

**设计原则**（遵循 pptx skill 规范）：
- 配色：选择主题相关的定制调色板，不默认为蓝色
- 每页必有视觉元素（图标、色块、图表），无纯文字页
- 字体：标题 Arial Black 36-44pt，正文 Calibri 12-16pt
- 布局多样化：卡片、双列、时间轴、表格交替使用
- 不使用 accent lines 作为标题下划线

## 关键决策点

### 时间范围
- 半年度汇报：6个月（如 2026-01 至 2026-06）
- 季度汇报：3个月
- 年度汇报：12个月
- 用户可能说"过去一年""今年上半年""Q2"等，需解析为具体日期

### 仓库来源
- 用户会提供 git 仓库 URL 或本地路径
- 支持多仓库合并（如算法仓库 + 工具仓库）
- 如果仓库已 clone 在 `/tmp` 下，先 `git fetch` 更新

### 节假日处理
- 自动跳过中国法定节假日（春节、国庆、五一、清明、端午、中秋）
- 处理调休工作日（节前/节后的周六日上班）

### 无提交日推理
- 原则：一个工作日没有 git commit 不代表没有工作
- 策略：根据前一个和后一个有提交日的工作内容进行推断
- 标注：生成的日报中明确标记"⚠️ 当日无 git 提交，内容由前后提交推理生成"

| `scripts/generate_reports.py` | Python | 阶段②③：从 git log 生成日报周报 |

## 使用示例

用户："请根据我过去半年的 git 提交记录生成一份工作汇报 PPT"

执行流程：
1. 确认时间范围（"过去半年"→ 2026-01-01 至今）
2. 确认仓库地址（用户提供或从上下文推断）
3. Clone/fetch 仓库 → 提取 git log
4. 运行 `generate_reports.py` → 生成 日报 + 周报
5. 分析周报内容 → 撰写汇报文档 → 设计 PPT 大纲
6. 根据大纲编写 PPTX 生成脚本 → 运行生成 → QA 检查
7. 输出 .pptx 文件路径

## 注意事项

1. **中文引号问题**：JS 字符串中的中文双引号 `""` 会导致解析错误，需使用单引号包裹或 Unicode 转义
2. **颜色格式**：pptxgenjs 不支持 `#` 前缀的 hex 颜色，必须使用纯 6 位 hex
3. **影子选项**：每次调用需用工厂函数创建新的 shadow 对象，复用会导致文件损坏
4. **breakLine**：多行文本必须在每项（除最后一项）设置 `breakLine: true`
5. **回避 Linux 专属命令**：Windows 环境避免使用 `sed`、AF_UNIX socket 等
