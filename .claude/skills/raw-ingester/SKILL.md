# raw-ingester — 原始数据摄入与需求盘问

你是 4 阶段工作流中「发现 + 盘问」阶段的执行者。你的任务是将 raw/ 文件夹中散落的原始数据，通过结构化的盘问环节，转化为清晰的产出需求，然后路由到对应的执行技能。

遵循 **一次一步** 原则，每次只展示一个步骤，等待用户回应后再继续。

---

## 一、资产组织原则

与本技能相关的目录分工：

| 目录 | 用途 |
|------|------|
| `raw/` | 原始数据暂存区，含未加工的文件（CSV/JSON/文本/PDF/图片等） |
| `7-Sources/` | 已加工来源，归档处理后的原始文件 |
| `8-Templates/` | 模板文件 |

---

## 二、处理流程

### 步骤 1/4 — 扫描 raw/ 发现新文件

```
📂 扫描 raw/ 文件夹...

→ raw/{子目录}/ 中有 {N} 个文件：

   {文件列表，按目录分组}
   · {文件名} — {文件类型} — {大小} — {最后修改时间}
   
   (如果没有文件 → 告知用户 raw/ 为空，建议先放入文件)

→ 这些文件是要一起处理，还是逐项处理？
```

如果用户有多个文件，先确认使用场景。如果文件属于同一项目/主题，建议一起处理。

### 步骤 2/4 — 逐项 Grill（分步盘问）

对每个文件或同组文件，按以下场景逐题提问。

**Grill 核心原则**：
- 每次只问 1 个问题，等用户回答后再问下一个
- 根据用户的回答动态调整后续问题
- 不要一次展示所有问题
- 用户可随时说"跳过"使用默认值

#### 通用问题（所有场景先问）

```
📋 关于这个文件「{文件名}」：

Q1: 这个文件是关于哪个项目或主题的？
    → （用户输入，用于确定 domain 和归档路径）

Q2: 你希望用它来产出什么？
    → A. 日报/周报/报告
    → B. 知识笔记（概念/来源/MOC/项目笔记）
    → C. 仅归档为参考资料
```

根据 Q2 的答案进入对应分支。

---

#### 分支 A：文件 → 日报/周报/报告

```
📋 关于产出报告：

Q3: 是日报、周报还是专项报告？
    → 日报 / 周报 / 专项报告

Q4: 日期范围是什么？
    → （如: 2026-05-29, 或 5月25-29日）

Q5: 是否需要包含文件中的原始数据表格？
    → 要 / 不要 / 只提取关键数据

Q6: 目标读者是谁？
    → 个人 / 团队 / 上级 / 客户

Q7: 有没有特殊格式或内容要求？
    → （用户输入，可选）

Q8: 报告标题是什么（可选，默认用日期+主题）？
    → （用户输入，可选）
```

---

#### 分支 B：文件 → 知识笔记

```
📋 关于产出笔记：

Q3: 要写什么类型的笔记？
    → A. 概念笔记 — 解释某个术语/知识点（→ 4-Resources/）
    → B. 来源笔记 — 记录外部参考资料（→ 7-Sources/）
    → C. MOC 索引 — 知识地图（→ 1-Atlas/）
    → D. 项目笔记 — 项目相关（→ 2-Projects/）

Q4: 所属领域（domain）是什么？
    → 如: tech/ai、life/grooming、business/management

Q5: 需要链接到 vault 中哪些已有笔记？
    → （用户输入已知的相关笔记，可选）

Q6: 笔记深度？
    → ⚡ 快速 — 简明要点，存摘要即可
    → 📝 标准 — 完整结构笔记
    → 🧬 原子化 — 拆解为多篇概念笔记

Q7: 期望存放在哪个路径？
    → （自动推荐，可修改。如: 4-Resources/{domain}/{概念名}.md）
```

---

#### 分支 C：文件 → 仅归档

```
📋 关于归档：

Q3: 归属于哪个项目或主题？
    → （用于确定 7-Sources/ 下的目标子目录）

Q4: 归档时是否需要同时创建一篇简要的来源笔记？
    → 要 — 创建一篇简短来源笔记记录文件来源和用途
    → 不 — 仅移动文件，不创建笔记
```

---

### 步骤 3/4 — 确认汇总

盘问完成后，展示汇总并请求确认：

```
📋 需求确认汇总

| 项目 | 值 |
|------|-----|
| 原始文件 | {文件名} |
| 产出类型 | {日报/笔记/归档} |
| 项目/主题 | {项目名} |
| 领域 | {domain} |
| 日期范围 | {日期} |
| 深度 | {深度} |
| 目标路径 | {路径} |

👉 确认以上信息正确？按 Enter 继续，或告诉我哪里需要修改。
```

### 步骤 4/4 — 路由到执行阶段

确认后，根据产出类型路由：

| 产出类型 | 路由目标 | 传递信息 |
|----------|----------|----------|
| 日报/周报/报告 | `daily-concierge`（1.3 节） | 文件路径、主题、日期、格式要求 |
| 概念笔记/来源笔记 | `info-digester` 或 `note-composer` | 文件路径、笔记类型、domain、链接要求 |
| 仅归档 | `workflow-wrapup` | 文件路径、目标项目 |

路由时向目标技能传递结构化的需求信息，确保后续阶段无需重复盘问。

完成后提示：
```
✅ 已路由到 {技能名} 执行。如有需要可随时中断或调整。
```

## 三、快捷模式

有经验的用户可直接提供完整信息跳过盘问：

```
/raw-ingester 把 raw/项目周报/数据.csv 创建为周报，项目:热轧排程，日期:5月25-29日
```

此时直接汇总确认后路由，不逐题盘问。

---

## 四、边界情况

- **raw/ 为空**：告知用户 raw/ 中没有文件，提示先放入文件或提供其他入口
- **未知文件类型**：提示用户文件格式无法识别，询问是否仍要尝试处理
- **多文件混合场景**：如果 raw/ 中有分属不同项目的文件，建议用户分批处理
- **中断恢复**：用户说"算了"或"不做了"，清理临时状态并返回 vault-wizard 菜单
