# my-skills 开发工作流技能体系

> 一句话定义：一套覆盖从项目启动到收尾全生命周期的 Claude Code 技能链，通过 11 个结构化技能串联需求分析、设计、编码、审查、调试、版本管理和知识整理。

## 概述

**my-skills** 是一个面向 Python 开发的 Claude Code 技能集合，作者 617luohe。它将软件开发流程拆解为 **10 个标准化阶段**，加上一个智能调度入口，形成完整的端到端工作流。

- **GitHub 仓库**：https://github.com/617luohe/my-skills
- **技能总数**：11 个（1 个调度器 + 10 个阶段技能）
- **覆盖范围**：项目初始化 → 规划 → 分析 → 原型 → 编码 → 审查 → 优化 → 调试 → 版本管理 → 收尾

## 工作流全景图

```
use-skills（智能调度器 — 自然语言入口）
    │
    ▼
0-启动  ── 新项目初始化（脚手架 + 环境 + 工具链）
    │
    ▼
1-规划  ── 方案设计（追问 → 术语 → 接口 → PRD → 任务拆解）
    │
    ├──→ 2-分析（代码理解与概览，不熟悉的代码先读地图）
    │
    ├──→ 3-原型（快速原型验证，不确定的决策先试错）
    │
    ▼
4-开发  ── TDD 编码（红-绿-重构，pytest 驱动）
    │
    ├──→ 5-检查（代码审查 + Bug 报告）
    │
    ├──→ 6-优化（架构分析 + 重构计划）
    │
    ├──→ 7-调试（六阶段结构化调试）
    │
    ▼
8-版本管理 ── Git 操作（保存/日志/分支/远程）
    │
    ▼
9-最后整理 ── 收尾交接（交接文档 + 知识同步 + 安全确认）
```

各个技能之间不是严格的线性关系 —— 实际使用时可跳转、回溯、跳过：

- **线性路径**：`0-启动 → 1-规划 → 4-开发 → 5-检查 → 8-版本管理 → 9-最后整理`
- **需要分析先走 2-分析**：`2-分析 → 6-优化` 或 `2-分析 → 4-开发`
- **有不确定性先走 3-原型**：`1-规划 → 3-原型 → 4-开发`
- **发现 Bug 走 7-调试**：`7-调试 → 5-检查` 或 `7-调试 → 4-开发（回归测试）`

## 技能详述

### use-skills — 智能技能调度器（入口）

**一句话**：不用记技能名称，说需求自动匹配。

**触发方式**：`/use-skills <你的需求>`

**工作流程**：
1. **理解需求** — 提取主要意图、次要意图、目标对象
2. **匹配技能** — 对照路由表找到最合适的技能，多技能按依赖排序
3. **展示计划** — 确认后再执行
4. **执行技能** — 严格按照对应 SKILL.md 的指令执行

**路由表示例**：

| 你说 | 匹配 |
|------|------|
| "新项目叫 data-pipeline" | 0-启动 |
| "规划一个登录功能，然后用 TDD 实现" | 1-规划 → 4-开发 |
| "分析这段代码然后重构" | 2-分析 → 6-优化 |
| "验证这个数据模型，没问题就实现" | 3-原型 → 4-开发 |

### 0-启动 — 新项目初始化

**一句话**：新建 Python 项目一条龙，从头搭建标准项目骨架。

**触发方式**：`/0-启动`

**完整流程**：
1. **确认项目信息** — 项目名称、Python 版本、mypy strict、Ruff 风格（行长度/引号风格）
2. **检测并安装 uv** — 自动安装（如果缺失）
3. **创建目录结构** — `src/project_name/` + `tests/` + `.gitignore` + `pyproject.toml`
4. **生成入口文件** — `main.py` + 占位测试
5. **配置 pyproject.toml** — Ruff / mypy / pytest 配置
6. **配置 pre-commit** — Ruff format + Ruff check + mypy + pytest
7. **创建 uv 虚拟环境** — `uv venv` + `uv add --dev pre-commit ruff mypy pytest`
8. **安装 pre-commit 钩子**
9. **初始化 git 仓库** — `git init` + 首次提交
10. **配置任务管理** — 本地 `.scratch/todo.md` 或 GitHub Issues

**典型对话**：
```
你：新建一个项目叫 data-pipeline，帮我启动
Claude：Python 版本？≥3.12？用 uv？
你：对
Claude：mypy 要 strict 吗？(y/N)
...
Claude：项目已就绪。运行 `uv run pytest` 验证。
```

### 1-规划 — 方案设计与任务拆解

**一句话**：写代码前系统性地理清需求、设计接口、输出 PRD、拆解任务。

**触发方式**：`/1-规划`

**五个阶段（按顺序推进）**：

| 阶段 | 内容 | 产出 |
|------|------|------|
| 1. 方案追问 | 决策树式穷举追问，暴露盲区 | 达成共识的方案描述 |
| 2. 领域术语 | 提取并统一术语表 | `UBIQUITOUS_LANGUAGE.md` |
| 3. 接口设计 | 3 个并行 Agent 生成多方案对比 | 选定接口方案 |
| 4. 输出 PRD | 结构化需求文档 | PRD 文档 |
| 5. 拆解任务 | 垂直切片，标注 AFK/HITL | 按依赖排序的 issue 清单 |

**核心原则**：`RELENTLESS` — 沿着依赖链逐个决议，不把所有分支走完不罢休。每个阶段结束时询问是否继续下一阶段。

**典型对话**：
```
你：我想给系统加一个缓存层，帮我规划一下
Claude：好的，先从最基本的开始——你要缓存什么数据？
       我建议缓存数据库查询结果...
       ...
[经过 5 个阶段]
Claude：规划完成，共拆出 4 个 issue：
       1. [AFK] 缓存基础设施
       2. [AFK] 查询缓存接入
       3. [AFK] 失效策略
       4. [HITL] 缓存监控
```

### 2-分析 — 代码理解与概览

**一句话**：遇到不熟悉的代码，跳出细节，输出一张架构地图。

**触发方式**：`/2-分析`

**输出四部分**：
1. **模块地图** — 涉及哪些模块/文件，各自职责
2. **数据流** — 数据如何流转：谁调谁、输入输出
3. **关键入口** — 从哪里开始读最有效率
4. **架构位置** — 这片代码在整个项目中的位置

**使用场景**：接手不熟悉的模块、看 PR 前了解全貌、新项目快速建立心智模型。

**典型对话**：
```
你：这个 views.py 完全看不懂，帮我分析一下
Claude：这是订单模块的视图层：
       User → OrderViewSet
         ├── list() → OrderSerializer → OrderRepository
         └── create() → PaymentService.process()
       
       建议从 OrderViewSet.create() (views.py:42) 开始读。
```

### 3-原型 — 快速原型验证

**一句话**：用一次性代码回答一个设计问题，验证不确定的决策。

**触发方式**：`/3-原型`

**两个分支**：

**分支 A：逻辑/状态原型（CLI 脚本）**
- 封装可移植逻辑模块（状态机、reducer、纯函数）
- 构建最小 TUI 外壳用于交互验证
- 规则：标记 `PROTOTYPE`、一条命令可跑、不持久化、不写测试、不打磨

**分支 B：UI/交互原型（Web）**
- 2-3 个截然不同的视觉方案
- 参数切换变体

**完成后**：记录结论 → 删除原型或吸收到正式代码 → 进入 4-开发

**反模式**：不写测试、不连真实数据库、不泛化、TUI 外壳不合入正式代码。

**典型对话**：
```
你：帮我验证一下这个订单状态机设计
Claude：你的设计中 refunded 状态还能再退款吗？
       先写个原型跑一下看看...
       State: pending → pay() → paid
       State: paid → refund() → refunding
       State: refunded → refund() → Error: 已退款不可再退款
       验证通过。
```

### 4-开发 — TDD 编码实现

**一句话**：pytest 驱动的红-绿-重构循环，四条编程准则贯穿全程。

**触发方式**：`/4-开发`（注意 SKILL.md 中写的是 `/3-开发`，可能为笔误，应以实际部署为准）

**四条编程准则**：
1. **先想后写** — 编码前说出假设，列出多方案
2. **简单第一** — 最少代码解决问题
3. **手术刀式改动** — 只碰必须碰的
4. **目标驱动执行** — 模糊任务转可验证目标

**TDD 流程**：
1. **规划** — 确认接口，列出测试行为，等你批准
2. **示踪弹** — 一个测试 + 最少代码通过
3. **递增循环** — RED（写测试确认失败）→ GREEN（最少代码通过），一次一个
4. **重构** — 全部变绿后提取重复、加深模块

**核心原则**：测试通过**公共接口验证行为**，不验证实现细节。**绝不在 RED 时重构**。

**典型对话**：
```
你：用 TDD 实现计算器 add 方法
Claude：接口确认：def add(a: int, b: int) -> int
       RED: test_add_returns_sum → 失败
       GREEN: def add(a, b): return a + b → 通过
```

### 5-检查 — 代码审查与验收

**一句话**：双轴并行审查代码质量，交互式提交 Bug 报告。

**触发方式**：`/5-检查`

**模式 A — 代码审查**：
- 双轴并行，用子代理独立执行：
  - **Standards 轴** — 命名规范、类型注解、异常处理、import 组织、文档字符串
  - **Spec 轴** — 需求符合度、范围蔓延、实现质量
- 汇总报告不合并不排序

**模式 B — Bug 报告**：
- 交互式 QA：聆听 → 澄清（最多 2-3 问）→ 探索代码 → 判断是否拆分 → 提交 GitHub Issue
- Issue 规则：不包含文件路径（会过期）、使用领域术语、描述行为而非代码、30 秒内读完

**典型对话**：
```
你：帮我检查一下 payment-service 分支
Claude：Standards 审查：
       ❌ payment/processor.py:45 — processPayment → process_payment
       Spec 审查：
       ❌ 缺失需求: PRD §3.2 "退款原路返回" 未实现
```

---

### 6-优化 — 重构与架构改进

**一句话**：发现架构摩擦点，制定渐进式重构方案。

**触发方式**：`/6-优化`

**模式 A — 架构分析**：
- 核心概念：深度（深模块好）、浅模块（坏）、接缝、删除测试
- 探索代码 → 展示候选问题 → 追问循环

**模式 B — 重构计划**：
- 听取描述 → 验证 + 检查测试覆盖 → 探讨选项 → 敲定范围 → 拆解为极小提交
- 每步尽可能小，始终可工作

**典型对话**：
```
你：payment_service.py 800 行了，帮我优化
Claude：支付、退款、对账、通知四个功能耦合在一起。
       建议分 5 步提取...每一步独立可提交。
```

### 7-调试 — 结构化调试

**一句话**：六阶段系统化排查难复现的 bug 和性能问题。

**触发方式**：`/7-调试`

**六阶段流程**：

| 阶段 | 核心任务 | 关键原则 |
|------|---------|---------|
| 1. 构建反馈回路 | 建立快速 pass/fail 信号 | **最重要的阶段**，投入不成比例的努力 |
| 2. 复现 | 确认回路产生正确失败模式 | 多次运行确认 |
| 3. 假设 | 列出 3-5 条可证伪假设 | 不能只有一个（锚定偏见） |
| 4. 工具验证 | 一次改一个变量验证假设 | 优先调试器/REPL |
| 5. 修复+回归 | 最小化复现 → 修复 → 回归测试 | 回归测试命名 `test_regression_<编号>` |
| 6. 清理 | 确认不复现、回归测试通过、调试标签清除 | 问"什么能防止这个 bug？" |

**反馈回路优先级**：失败测试 → HTTP 脚本 → CLI 复现 → 回放捕获 → 一次性 harness → 模糊循环 → 二分 harness → 差异对比

**典型对话**：
```
你：订单支付接口偶尔返回 500
Claude：先构建反馈回路——curl 循环调用 100 次看失败率。
       发现约 5% 失败。3 条假设：
       1. 连接池耗尽 2. 网关超时 3. 缓存穿透
```

### 8-版本管理 — Git 版本控制

**一句话**：git 全流程操作，默认本地仓库，按需连接 GitHub。

**触发方式**：`/8-版本管理`

**操作清单**：

| 操作 | 命令 | 说明 |
|------|------|------|
| init | `git init` | 初始化仓库 |
| save | `git add . + commit` | 保存版本，自动生成提交信息 |
| log | `git log --oneline --graph` | 查看历史 |
| diff | `git diff` | 查看变更 |
| rollback | `git revert` | 安全回滚（默认，保留历史） |
| reset | `git reset --hard` | 硬重置（危险，需明确同意） |
| branch | `git branch / checkout` | 分支管理 |
| remote | `git remote add` | 连接 GitHub |
| push/pull | `git push / pull` | 同步远程 |

**典型对话**：
```
你：帮我保存一下进度
Claude：git commit -m "feat: 完成用户模块基础功能"
你：看看改了什么
Claude：git log --oneline --graph
你：推送到 GitHub
Claude：仓库 URL 是什么？
```

### 9-最后整理 — 收尾与交接

**一句话**：阶段性收尾时做三件事：交接文档、知识同步、安全确认。

**触发方式**：`/9-最后整理`

**三步骤**：

**1. 会话交接** — 创建 `handoff-xxx.md`，引用已有产物而非重复内容，建议下个会话用到的技能

**2. 知识整理** — 同步三层知识体系：
| 位置 | 职责 |
|------|------|
| Agent 记忆系统 (`~/.claude/projects/.../memory/`) | 跨会话复用 |
| CLAUDE.md | 下次 AI 会话的规则手册 |
| docs/ + README.md | 面向他人的文档 |

**3. 安全护栏确认** — 检查 PreToolUse 钩子是否配置生效

**典型对话**：
```
你：今天做完了，帮我整理一下
Claude：Handoff → handoff-a7k3.md
       知识同步 → CLAUDE.md 已更新
       安全护栏 → 已确认生效
```

## 使用方式

### 部署到 Claude Code

```bash
# 克隆仓库
git clone https://github.com/617luohe/my-skills.git

# 将需要的技能复制到 .claude/skills/
# 例如：
cp -r my-skills/use-skills ~/.claude/skills/
cp -r my-skills/1-规划 ~/.claude/skills/
cp -r my-skills/4-开发 ~/.claude/skills/

# 也可以使用 Skills Manager 管理
```

### 推荐使用路径

| 场景 | 技能组合 | 说明 |
|------|---------|------|
| **全流程新项目** | 0-启动 → 1-规划 → 4-开发 → 5-检查 → 8-版本管理 | 从零到提交 |
| **接手遗留代码** | 2-分析 → 6-优化 → 4-开发 → 5-检查 | 先理解再改进 |
| **修复 Bug** | 7-调试 → 4-开发 → 5-检查 → 8-版本管理 | 结构化排查修复 |
| **快速原型验证** | 3-原型 → 4-开发 | 试错后正式实现 |
| **不知道用哪个** | use-skills | 自然语言自动路由 |

### 最佳实践

- **按阶段推进，不跳跃** — 1-规划 的 5 个阶段按顺序执行，不要跳过
- **不确定就原型** — 任何不确定的决策都可以走 3-原型 试错
- **先分析再优化** — 6-优化 前建议先走 2-分析 了解全貌
- **检查是双向的** — 5-检查 既审查代码规范，也审查需求符合度
- **反馈回路是关键** — 7-调试 中最重要的是阶段 1（构建反馈回路），投入不成比例的努力
- **收尾必做** — 9-最后整理 确保下次会话能无缝继续
