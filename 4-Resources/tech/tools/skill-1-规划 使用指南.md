# 1-规划 — 方案设计与任务拆解

> 一句话：写代码前，用 6 个阶段把需求追问到底、统一术语、设计接口、输出 PRD、拆解成可独立执行的任务。

## 什么时候用

| 场景 | 说明 |
|------|------|
| 中型以上功能开发 | 涉及多个模块、需要前后端配合、有复杂业务逻辑 |
| 方案不确定 | "要不要用消息队列？""同步还是异步？"——需要系统性梳理 |
| 团队协作 | 需要产出一份 PRD 和任务清单给团队成员 |
| 接手一个模糊需求 | 需求方只说了"做用户系统"，其他都是空白 |

**不适用**：修一个 typo、改一个配置、加一行验证逻辑——这些直接用 4-开发。

## 6 阶段流程

```
阶段 1 ── 方案追问（★★★ 核心 ★★★）
  │ 扫描已有文档 → 决策树穷举追问 → 所有分支决议才退出
  │
阶段 2 ── 领域术语提取
  │ 统一术语表 → 实时更新 CONTEXT.md → 标记歧义
  │
阶段 2.5 ── 决策记录（ADR）
  │ 仅记录难逆转 + 少见 + 有真实取舍的决策
  │
阶段 3 ── 接口设计
  │ 3 个并行 Agent 生成不同方案 → 对比 → 选定
  │
阶段 4 ── 输出 PRD
  │ 基于共识综合结构化需求文档，不追问用户
  │
阶段 5 ── 拆解任务
    垂直切片（schema → 逻辑 → 测试）→ 标注 AFK 或 HITL
```

## 案例演示

### 案例 1：订单取消功能（完整 6 阶段）

```
用户：/1-规划 帮我想想怎么做订单取消功能

Claude：luohe，开始规划。

       === 阶段 1：方案追问 ===

       [文档发现] 扫描项目...
       发现已有 CONTEXT.md，其中定义 Order 状态包括 PENDING、PAID、REFUNDING。
       CONTEXT.md 定义"取消"是用户主动撤销未支付订单。
       
       但你说"取消"——是不是也包括已支付订单的取消？

用户：两种都要支持

Claude：那是两个不同的流程。
       
       未支付取消：直接关单，不退钱（因为没付）。
       已支付取消：触发退款流程，走 REFUNDING → REFUNDED。
       
       已支付订单取消后，退款怎么处理？
       我推荐自动原路退款 + 异步确认。你觉得呢？

用户：可以

Claude：[场景压力测试] 如果一个订单分多次发货，部分发货后取消——
       已发货部分是否要单独退款？还是整个订单全部退回？

用户：部分发货不允许取消，整单取消才行

Claude：明白了。再加一个场景：订单已经进入配送状态，用户还能取消吗？

用户：不能，配送中只能申请退货，不是取消

Claude：好。总结：取消 = (未支付取消) OR (已支付且未发货取消)。两个分支。
       
       [✓ 所有决策分支已决议]
       
       === 阶段 2：领域术语 ===
       
       术语表：
       | 术语 | 定义 | 避免用 |
       |------|------|--------|
       | 取消（Cancel） | 用户主动终止未发货订单 | 撤销、作废 |
       | 退款（Refund） | 已支付订单取消后的资金退回 | 打款、返款 |
       | 退货（Return） | 已发货后用户寄回商品 | 退换货 |
       
       [已更新 CONTEXT.md]

       === 阶段 2.5：ADR ===
       
       有一项满足 ADR 三个条件（难逆转 + 少见 + 真实取舍）：
       
       ADR-0003: 已支付取消走异步退款
       订单取消接口只负责创建退款单并返回，实际退款由后台异步处理。
       原因：支付网关回调可能超时 5-10s，同接口内等待会阻塞用户。
       备选方案：同步等待——被否决，用户体验极差。
       
       === 阶段 3：接口设计 ===

       [并行 Agent 生成 3 个方案]
       
       方案 A（最小接口）：
       POST /orders/{id}/cancel → 202 Accepted
       
       方案 B（灵活接口）：
       POST /orders/{id}/cancel
         body: { reason?: string, refund_method?: "original" | "wallet" }
       
       方案 C（场景优化）：
       同上，外加 GET /orders/{id}/cancel-eligibility 预检接口
       
       选定：方案 A + 预检接口（方案 C）——简洁但保留扩展性。
       
       === 阶段 4：PRD ===
       
       # 订单取消功能 PRD
       ## Problem Statement
       用户无法取消已下但未发货的订单...
       ## User Stories
       - As a buyer, I want to cancel unpaid order...
       - As a buyer, I want to cancel paid but unshipped order...
       ...
       
       === 阶段 5：任务拆解 ===
       
       共拆出 5 个 issue（按依赖排序）：
       
       1. [AFK] 订单状态机扩展 — OrderState 新增 CANCELLING/CANCELLED
       2. [AFK] 取消资格预检 — GET /orders/{id}/cancel-eligibility
       3. [AFK] 未支付取消 — POST /orders/{id}/cancel（PENDING→CANCELLED）
       4. [HITL] 已支付取消+退款 — POST /orders/{id}/cancel（PAID→CANCELLING→退款）
       5. [AFK] 取消日志 — 所有取消操作的审计日志
       
       规划完成。
```

### 案例 2：轻量级使用——只做到阶段 1

```
用户：/1-规划 我想加一个缓存层，不确定用什么方案

Claude：luohe，开始规划。

       === 阶段 1：方案追问 ===

       [文档发现] 未找到 CONTEXT.md，阶段 2 会创建。

       先从最基本的开始——你要缓存什么？
       我建议缓存数据库查询结果，而不是计算中间结果。你觉得？

用户：对，查库结果

Claude：缓存粒度？
       A) 全表缓存 — 简单但浪费内存
       B) 逐行缓存 — 命中率低
       C) 按查询参数缓存 — 命中率最高
       
       我推荐 C，按查询参数缓存。

用户：选 C

Claude：过期策略？
       A) TTL（固定时间过期）
       B) 写入失效（数据更新时清缓存）
       C) TTL + 写入失效
       
       C 是最常见的组合。你觉得？

用户：用 C，TTL 设 5 分钟

Claude：[场景压力测试] 如果一个用户更新了数据，要不要清掉所有用户的缓存？还是只清自己的？
...

[方案追问完成后]

Claude：方案已达成共识。是否继续阶段 2（领域术语 + CONTEXT.md）？

用户：不用了，这就够了。直接开始写代码吧

Claude：好的。方案摘要：
       - 缓存层：按查询参数缓存数据库查询结果
       - 过期策略：TTL 5min + 写入失效
       - 存储：Redis
       - 接口：装饰器模式 @cached(ttl=300)
       
       可以进入 4-开发 开始 TDD 实现了。
```

## 注意事项

- 阶段 1 的核心原则是 **RELENTLESS**——不把所有决策分支走完不罢休
- 每个阶段结束时问你是否继续，你可以随时停止
- ADR（阶段 2.5）**不是所有决策都记录**——只记录满足三个条件的
- 可以跳过不需要的阶段（比如小型功能跳过阶段 3 接口对比）
- CONTEXT.md 是**内联实时更新**的，不是最后批量写
