# Obsidian 知识库体系设计

## 一、设计哲学

### 核心理念：AI-Native 知识管理

传统知识库为人而建；AI 时代的知识库需要**人机双读**。每条笔记既是人的思考载体，也是 AI Agent 可理解、可操作的知识单元。

### 五条设计原则

| 原则 | 含义 | 实践 |
|------|------|------|
| **可发现性** | 任何内容都能通过搜索、链接或索引在 30 秒内找到 | MOC + 反向链接 + 标签体系 |
| **原子性** | 一条笔记只承载一个概念/想法 | 长笔记拆分为多条原子笔记 + 一条 MOC |
| **可溯源** | 每条知识都能追溯到其来源 | 来源笔记 + `source` 属性 |
| **渐进式完善** | 笔记从草稿到成熟逐步进化 | seed → sprout → evergreen 置信度分级 |
| **可计算性** | AI Agent 能够解析、操作、链接笔记 | 标准化 frontmatter + 清晰的结构约定 |

## 二、目录结构设计

```
Obsidian Vault/
├── 0-Inbox/              ← 捕获区：未经处理的输入
├── 1-Atlas/              ← 知识图谱的总索引（MOC）
├── 2-Projects/           ← 有截止日期的项目
├── 3-Areas/              ← 无截止日期的责任域
├── 4-Resources/          ← 按主题组织的知识库
├── 5-Journal/            ← 时间维度记录（日/周/月）
├── 6-People/             ← 人物与关系
├── 7-Sources/            ← 外部来源（书/文章/课程）
├── 8-Templates/          ← 笔记模板
├── 9-System/             ← 知识库管理与技能规范
└── assets/               ← 二进制资源（图片/PDF/音频）
```

### 0-Inbox —— 捕获区

**用途**：所有零散想法、临时记录、待处理素材的唯一入口。

**内容类型**：
- 语音转文字片段
- 网页剪藏
- 随手记下的想法
- 会议现场笔记
- 待分类的资料

**命名规范**：`YYYYMMDD-HHMM-{简短主题}.md`（例如 `20260514-0930-AI知识库设计思路.md`）

**属性规范**：
```yaml
status: inbox          # inbox → processing → processed
captured: 2026-05-15   # 捕获时间
source:                # 来源（可选）
```

**处理节奏**：每日处理一次，目标 inbox zero。处理后的笔记要么转入对应目录，要么合并到已有笔记后删除。

### 1-Atlas —— 知识图谱骨架

**用途**：存放 Map of Content（MOC），即知识库的目录和导航中心。Atlas 是知识库的骨架——它不直接承载知识，而是组织和导航知识。

**内容类型**：
- 领域 MOC（如 `MOC-机器学习.md`、`MOC-项目管理.md`）
- 概念索引（如 `术语表.md`）
- 跨领域导航页
- 知识结构概览图

**命名规范**：`MOC-{领域名}.md` 或 `{领域}-索引.md`

**MOC 文件标准结构**：
```markdown
---
tags: [type/moc, domain/xxx]
created: YYYY-MM-DD
---

# {领域名}

## 核心概念
- [[概念A]] — 一句话描述
- [[概念B]] — 一句话描述

## 延伸阅读
- [[相关MOC-1]]

## 待探索
- [ ] 尚未覆盖的子主题
```

**关联关系**：
- Atlas → Resources：MOC 引用 Resource 中的原子笔记
- Atlas → Projects：MOC 引用相关项目笔记
- Atlas → Sources：MOC 引用相关来源笔记
- Atlas 之间：MOC 之间可以互相引用，形成层级结构

### 2-Projects —— 项目管理

**用途**：有明确目标和截止日期的工作。每个项目是一个独立文件夹。

**目录结构**：
```
2-Projects/
└── {项目名}/
    ├── {项目名}.md          ← 项目概述（目标、状态、时间线）
    ├── tasks/               ← 任务拆解
    ├── notes/               ← 项目相关笔记
    └── references/          ← 项目参考资料
```

**项目概述模板属性**：
```yaml
status: active | paused | completed
priority: P0 | P1 | P2 | P3
deadline: YYYY-MM-DD
started: YYYY-MM-DD
completed: null
tags: [type/project, domain/xxx]
```

**生命周期**：项目完成后，整体移至 `99-Archives/`。

**关联关系**：
- Project → Area：项目属于某个责任域
- Project → Resource：项目产出知识，丰富对应资源
- Project → People：项目涉及的相关人员
- Project → Source：项目引用的外部资料

### 3-Areas —— 持续责任域

**用途**：没有截止日期、需要持续关注的生活/工作领域。

**示例**：
- `3-Areas/健康/` — 运动、饮食、体检
- `3-Areas/财务/` — 投资、预算、税务
- `3-Areas/职业/` — 技能发展、职业规划
- `3-Areas/关系/` — 家庭、朋友、社交

**每个 Area 的入口文件属性**：
```yaml
category: area
review-frequency: weekly | monthly | quarterly
last-review: YYYY-MM-DD
tags: [type/area, domain/xxx]
```

**与 Projects 的区分**：
- Area 没有截止日期，Project 有
- Area 是"我要持续做好的事"，Project 是"我要完成的事"
- Area 产生 Project，Project 完成后沉淀进 Area 或 Resource

### 4-Resources —— 主题知识库

**用途**：按主题组织的原子化知识笔记。这是知识库的核心内容层。

**组织方式**：按领域建立子目录，例如：
```
4-Resources/
├── tech/
│   ├── ai/
│   ├── backend/
│   └── devops/
├── business/
│   ├── strategy/
│   └── marketing/
└── humanities/
    ├── psychology/
    └── philosophy/
```

**笔记置信度分级**：
```yaml
confidence: seed      # 刚种下的种子：初步想法，可能不完整
confidence: sprout    # 发芽：有一定内容，链接初步建立
confidence: evergreen # 长青：完善、链接丰富、经常维护
```

**原子笔记编写原则**：
- 一个文件只讲一个概念
- 第一段必须给出概念的简洁定义
- 使用 `[[wikilink]]` 而非孤立写作
- 区分"事实"和"观点"，明确标注

### 5-Journal —— 日记体系

**用途**：按时间维度组织记录、反思和追踪。

**文件层级**：
```
5-Journal/
├── daily/
│   └── 2026/
│       ├── 2026-05-15.md
│       └── ...
├── weekly/
│   └── 2026-W20.md
├── monthly/
│   └── 2026-05.md
└── quarterly/
    └── 2026-Q2.md
```

**日记属性**：
```yaml
date: 2026-05-15
tags: [type/journal, type/daily]
mood: 1-5
energy: 1-5
```

**日记→永久笔记的提炼管道**：
日记中的有价值内容，在每日或每周回顾时，提炼为永久笔记放入 Resources。日记是原料，永久笔记是成品。

### 6-People —— 人物网络

**用途**：记录与个人相关的人物信息、互动历史。

**每人一个文件**，使用属性记录结构化信息：
```yaml
aliases: [昵称, 英文名]
role: 同事 | 朋友 | 导师 | ...
organization: 公司/组织
relationship: 工作 | 私人 | 学习
last-contact: YYYY-MM-DD
tags: [type/person]
```

**关联关系**：
- People → Meeting Notes：会议笔记通过 wikilink 关联参会人
- People → Projects：项目涉及的人员
- People → Sources：某本书/文章的作者

### 7-Sources —— 来源管理

**用途**：管理所有外部知识来源，区分"来源信息"和"学习心得"。

**源类型**：
```yaml
type: book | article | paper | video | podcast | course | webpage
```

**来源笔记标准结构**：
```markdown
---
title: "书名/文章标题"
author: "作者"
source-type: book
url: "链接"
date-read: YYYY-MM-DD
rating: 1-5
tags: [type/source, type/book, domain/xxx]
---

# {标题}

## 元数据
- 作者：
- 出版日期：
- 阅读日期：

## 核心论点
1. 论点一
2. 论点二

## 关键概念
- [[概念A]]
- [[概念B]]

## 我的思考
（个人评注，区别于原文摘要）

## 行动项
- [ ] 基于本书的待办事项
```

**来源→概念→知识网络的流程**：
1. 读一本书 → 创建来源笔记（7-Sources）
2. 提取关键概念 → 创建原子笔记（4-Resources）
3. 原子笔记链接回来源 → 同时链接到相关 MOC（1-Atlas）

### 8-Templates —— 模板体系

**用途**：存放所有笔记模板，确保创建一致性。

**核心模板**：
```
8-Templates/
├── tpl-MOC.md              ← MOC 笔记模板
├── tpl-project.md          ← 项目概述模板
├── tpl-area.md             ← 责任域入口模板
├── tpl-resource.md         ← 原子知识笔记模板
├── tpl-daily.md            ← 每日笔记模板
├── tpl-weekly.md           ← 每周回顾模板
├── tpl-monthly.md          ← 每月回顾模板
├── tpl-person.md           ← 人物笔记模板
├── tpl-source.md           ← 来源笔记模板
├── tpl-meeting.md          ← 会议笔记模板
└── tpl-flashcard.md        ← 闪卡笔记模板
```

**用途**：存放知识库本身的元信息、规范和技能设计文档。

```
9-System/

## 三、元数据与标签体系

### 属性（Frontmatter）标准

所有笔记必须包含的核心属性：

| 属性 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `title` | string | 笔记标题（可选，默认取文件名） | `知识库体系设计` |
| `aliases` | list | 别名，用于链接时的不同称呼 | `[设计规范, spec]` |
| `tags` | list | 层级标签 | `[system/design, type/spec]` |
| `created` | date | 创建日期 | `2026-05-15` |
| `updated` | date | 最后更新日期（自动） | `2026-05-15` |
| `status` | string | 生命周期状态 | `inbox / draft / review / published / archived` |

按笔记类型附加属性：

| 笔记类型 | 附加属性 |
|----------|----------|
| Project | `priority`, `deadline`, `started`, `completed` |
| Area | `review-frequency`, `last-review` |
| Resource | `confidence`, `source`, `domain` |
| Journal | `date`, `mood`, `energy` |
| Person | `role`, `organization`, `last-contact` |
| Source | `author`, `source-type`, `url`, `date-read`, `rating` |

### 标签层级

```
#type/          ← 笔记类型
  moc           ← MOC 索引笔记
  project       ← 项目笔记
  area          ← 责任域笔记
  resource      ← 知识笔记
  journal       ← 日记
  person        ← 人物笔记
  source        ← 来源笔记
  template      ← 模板
  meeting       ← 会议笔记

#status/        ← 生命周期
  inbox         ← 待处理
  active        ← 进行中
  paused        ← 暂停
  completed     ← 已完成
  archived      ← 已归档

#confidence/    ← 笔记成熟度
  seed          ← 种子
  sprout        ← 发芽
  evergreen     ← 长青

#domain/        ← 知识领域（可自定义扩展）
  tech          ← 技术
  business      ← 商业
  science       ← 科学
  humanities    ← 人文
  health        ← 健康
  finance       ← 财务
```

## 四、技能体系设计

### 技能全景图

```
                    ┌──────────────┐
                    │ info-digester│ ← 外部信息→结构化笔记
                    └──────┬───────┘
                           │ 产生来源笔记和概念笔记
                           ▼
┌──────────────┐   ┌──────────────┐   ┌──────────────┐
│source-curator│──▶│note-composer │──▶│knowledge-linker│
│ 来源管理      │   │ 笔记撰写      │   │ 知识链接       │
└──────────────┘   └──────┬───────┘   └──────┬───────┘
                          │                   │
                          ▼                   ▼
                   ┌──────────────┐   ┌──────────────┐
                   │vault-auditor │◀──│journal-enhancer│
                   │ 知识库审计     │   │ 日记增强        │
                   └──────────────┘   └──────────────┘
```

六项技能覆盖四个核心能力维度：

| 维度 | 对应技能 |
|------|----------|
| 文档编写 | note-composer, journal-enhancer |
| 文档管理 | vault-auditor, knowledge-linker |
| 文档更新 | note-composer（更新模式）, vault-auditor（陈旧检测） |
| 信息提取 | info-digester, source-curator |

### Skill 1：note-composer（笔记撰写助手）

**标识符**：`note-composer`

**定位**：辅助用户撰写高质量、结构化、链接丰富的 Obsidian 笔记，覆盖从空白到成文的全流程。

#### 核心能力

| 能力 | 描述 |
|------|------|
| 模板匹配 | 根据用户意图自动选择最合适的笔记模板 |
| 内容生成 | 从大纲、要点、语音转录生成完整笔记 |
| Wikilink 推荐 | 分析正文内容，搜索 vault 中相关笔记并建议链接 |
| 属性补全 | 自动填写和验证 frontmatter，确保符合约定 |
| 格式规范 | 确保输出符合 vault 编写规范（标题层级、callout 使用、列表格式） |
| 更新模式 | 读取已有笔记，在不破坏结构的前提下插入/修改内容 |

#### 输入定义

```
Input:
  type: "create" | "update" | "expand"
  target:
    path: string?                    // 更新时必填，目标笔记路径
    note_type: "moc" | "project" | "area" | "resource" | "meeting" | "person" | "source" | "daily"
  content:
    raw: string                      // 用户的原始输入（大纲/想法/语音转录/片段）
    structure: "outline" | "draft" | "bullet" | "freeform"
  options:
    auto_link: boolean (default: true)   // 是否自动搜索并推荐链接
    auto_confirm: boolean (default: false) // 是否跳过确认直接写入
    template: string?                     // 指定模板路径（覆盖自动匹配）
```

```
Output:
  preview: string                    // 生成的笔记预览（Markdown）
  suggested_links:                   // 推荐的已有笔记链接
    - note_path: string
      relevance: "high" | "medium" | "low"
      context: string                // 为什么推荐这个链接
  metadata_report:
    properties_complete: boolean
    warnings: string[]
  result_file: string?               // 写入后的文件路径
```

#### 工作流

```
用户输入 → [类型识别] → [加载模板]
                          ↓
                    [搜索相关笔记]
                          ↓
                    [生成内容草案]
                          ↓
                    [检查属性完整性]
                          ↓
                    [展示预览 + 链接建议]
                          ↓
                    [用户确认] → [写入文件]
```

- **类型识别**：分析用户输入，判断笔记类型（MOC/项目/资源/会议等）。关键词匹配 + 上下文推断。
- **加载模板**：从 `8-Templates/` 加载对应模板。若模板不存在，使用内置默认结构。
- **搜索相关笔记**：调用 obsidian-cli 的搜索功能，基于正文关键词搜索 vault，返回按相关度排序的已有笔记列表。
- **生成内容草案**：基于模板和用户输入，生成完整 Markdown。确保 wikilink 语法正确。
- **检查属性完整性**：验证必填属性是否齐全，缺失项提示用户补充或自动填充默认值。
- **用户确认**：展示预览，允许用户修改后再写入。

#### 边界约束

- **不自动创建文件**：默认需用户确认后才写入。`auto_confirm: true` 时除外。
- **不删除或覆盖已有内容**：更新模式下，只添加或修改指定部分，保留其他内容不变。
- **搜索限定在 vault 内**：wikilink 推荐只搜索 vault 内已有笔记，不链接外部资源。
- **不确定事实标记**：对 AI 推测或不确定的内容，自动添加 `[需要验证]` 标记。
- **属性不覆盖手动填写**：如果笔记已有用户手动填写的属性，不自动覆盖，仅在缺失时补全。

#### 触发条件

用户消息匹配以下模式之一：
- "写一篇笔记关于..."
- "创建笔记..."
- "帮我整理成笔记..."
- "把这个大纲展开..."
- "更新 XXX 笔记..."
- "给 XXX 笔记补充..."

#### 使用示例

```
用户: "写一篇笔记关于 Retrieval-Augmented Generation，要放在 4-Resources/tech/ai/ 下"

Agent 行为:
1. 识别类型 → resource（原子知识笔记）
2. 加载 tpl-resource.md 模板
3. 搜索 vault 中已有的 AI 相关笔记 → 找到 [[LLM基础]]、[[向量数据库]]
4. 生成笔记，内容包含：
   - Frontmatter 属性
   - RAG 的定义
   - 核心原理
   - 与已有笔记的链接
   - [需要验证] 标记（如果有推测内容）
5. 展示预览，等待确认
6. 写入 4-Resources/tech/ai/RAG.md
```

```
用户: "更新 2-Projects/知识库搭建/知识库搭建.md，状态改为已完成"

Agent 行为:
1. 读取目标笔记
2. 识别更新意图 → 修改 status 属性 + 补充完成日期
3. 搜索 vault 中是否有遗漏的相关笔记需要链接
4. 展示 diff，等待确认
5. 写入更新
```

### Skill 2：knowledge-linker（知识链接引擎）

**标识符**：`knowledge-linker`

**定位**：维护知识库的链接网络，确保知识之间的关联被充分建立，减少信息孤岛。

#### 核心能力

| 能力 | 描述 |
|------|------|
| 孤岛检测 | 扫描 vault，列出没有任何入链的笔记 |
| 链接推荐 | 基于内容相似度，推荐笔记之间应建立链接 |
| MOC 完整性 | 检测 MOC 中是否遗漏了相关资源笔记 |
| 反向链接分析 | 分析某笔记的出链/入链比，评估链接健康度 |
| 失效链接修复 | 检测 [[wikilink]] 指向的文件是否存在，报告失效链接 |
| 链接图谱 | 为指定笔记生成局部链接关系图（用于可视化） |

#### 输入定义

```
Input:
  scope:
    mode: "global" | "targeted" | "moc"
    paths: string[]?                  // targeted 模式下指定的笔记列表
    moc_path: string?                 // moc 模式下指定的 MOC 文件
  analysis:
    type: "orphans" | "recommendations" | "moc-gap" | "health" | "broken"
    threshold: number (default: 0.5)  // 链接推荐的相似度阈值
  options:
    max_suggestions: number (default: 10)
```

```
Output:
  report:
    summary: string                   // 一句话总结
    findings:
      - type: string
        severity: "info" | "warning" | "critical"
        description: string
        suggestion: string
        source_note: string
        target_note: string?
    stats:
      total_notes: number
      orphan_count: number
      avg_links_per_note: number
      broken_links: number
```

#### 工作流

**孤岛检测**：
```
列出所有 .md 文件 → 对每个文件检查反向链接 → 筛选入链数=0 → 排除排除列表（模板、系统文件等） → 输出孤岛列表
```

**链接推荐**：
```
获取目标笔记 → 提取标题+标签+正文关键词 → 搜索 vault 中有相似关键词的笔记 → 排除已有链接 → 按相似度排序 → 输出推荐列表
```

**MOC 完整性**：
```
读取 MOC 文件 → 提取 MOC 的主题标签和关键词 → 搜索 4-Resources 中匹配但未被 MOC 引用的笔记 → 输出缺失列表
```

#### 边界约束

- **只读分析，不自动修改**：所有操作仅生成报告和建议，不写入任何文件。
- **排除系统文件**：不分析 `8-Templates/`、`9-System/`、`assets/` 下的文件。
- **尊重用户判断**：相似度只是建议，不强制链接；用户可能故意不链接某些笔记。
- **模板文件豁免**：模板文件天然缺少入链，不计入孤岛。

#### 触发条件

- "这个笔记和哪些相关？"
- "检查知识库链接健康"
- "找一下孤岛笔记"
- "更新 MOC XXX"
- "XXX MOC 里漏了什么？"

#### 使用示例

```
用户: "检查一下 1-Atlas/MOC-机器学习.md 的完整性"

Agent 行为:
1. 读取 MOC-机器学习.md
2. 提取其主题标签（#domain/tech/ai, #domain/tech/ml）
3. 搜索 4-Resources/tech/ai/ 下所有笔记
4. 对比 MOC 中已有的 wikilink 列表
5. 发现 3 篇笔记未被引用
6. 生成报告：
   - MOC-机器学习 当前引用 12 篇笔记
   - 缺失引用 3 篇：[[Transformer架构]]、[[迁移学习]]、[[模型评估指标]]
   - 建议将这 3 篇添加到对应章节
```

### Skill 3：info-digester（信息消化器）

**标识符**：`info-digester`

**定位**：将外部信息转化为结构化、可链接的知识笔记。是外部世界进入知识库的"消化管道"。

#### 核心能力

| 能力 | 描述 |
|------|------|
| 内容获取 | 从 URL、PDF、文本粘贴获取原始内容（调用 defuddle / pdf skill） |
| 类型识别 | 自动识别内容类型（文章/论文/书籍章节/视频字幕/播客转录） |
| 关键提取 | 提取核心论点、关键术语、重要案例、行动项 |
| 分层输出 | 根据深度参数输出：精简摘要 / 详细笔记 / 原子化拆解 |
| 增量合并 | 如果来源已有笔记，智能合并而非创建重复文件 |
| 元数据捕获 | 自动提取标题、作者、发布日期、来源 URL |

#### 输入定义

```
Input:
  source:
    type: "url" | "file" | "text"
    data: string                      // URL、文件路径、或原始文本
  depth:
    level: "summary" | "detailed" | "atomic"
    // summary: 300字以内摘要
    // detailed: 结构化完整笔记
    // atomic: 拆解为多条原子笔记 + 一条索引
  target:
    vault_path: string                // 存放位置（默认 7-Sources/）
    tags: string[]                    // 附加标签
  options:
    create_source_note: boolean (default: true)
    link_existing: boolean (default: true)  // 是否链接已有笔记
    language: "auto" | "zh" | "en"
```

```
Output:
  source_note_path: string            // 来源笔记路径
  concept_notes: string[]             // 创建的原子笔记路径列表
  summary: string                     // 消化摘要
  key_concepts:                       // 提取的关键概念
    - term: string
      definition: string
      linked_note: string?            // 对应的已有笔记
  action_items: string[]              // 提取的行动项
```

#### 工作流

```
[获取内容] ← 调用 defuddle (URL) / pdf skill (PDF) / 直接解析 (文本)
     ↓
[类型识别] ← 分析内容结构判断类型
     ↓
[提取元数据] ← 标题、作者、日期等
     ↓
[创建来源笔记] → 写入 7-Sources/（使用 tpl-source 模板）
     ↓
[内容分析] ← 按 depth 层级提取
     ↓
[概念链接] ← 搜索 vault 中已有相关笔记
     ↓
[输出结果] → 来源笔记 + 多篇原子笔记（atomic 模式）+ 摘要
```

#### 深度参数说明

**summary 模式**：
- 输出：300 字以内结构化摘要
- 适用：快速判断内容价值，决定是否深入阅读

**detailed 模式**：
- 输出：一篇完整的结构化笔记
- 结构：核心论点 → 支撑论据 → 关键案例 → 个人思考 → 行动项

**atomic 模式**：
- 输出：一篇来源索引笔记 + N 篇原子概念笔记
- 每篇原子笔记聚焦一个概念，独立可链接
- 适合高价值内容，需要深度整合进知识库

#### 边界约束

- **不绕过付费墙**：如果 URL 内容需要登录/付费，返回错误提示而非尝试绕过。
- **保留出处**：所有提取内容必须标注原始来源链接。
- **置信度标记**：AI 的解读和总结与原文引用需明确区分。使用 `> [!ai-interpretation]` callout 标记 AI 解读部分。
- **不消化未授权内容**：用户必须明确提供 URL/文件/文本，不主动搜索外部内容。
- **长内容分段处理**：超过 8000 字的内容分段消化，每段独立处理后再合并。
- **版权意识**：不存储原文全文（除非用户明确要求），只保留摘要和概念笔记。

#### 触发条件

- "消化这篇文章..."
- "把这个网页转成笔记..."
- "分析这篇论文..."
- "把这份 PDF 整理成笔记..."
- "处理这个视频的字幕..."

#### 使用示例

```
用户: "消化这篇文章：https://example.com/ai-trends-2026，depth=atomic"

Agent 行为:
1. 调用 defuddle 获取文章正文
2. 识别为技术趋势文章
3. 创建来源笔记：7-Sources/2026-AI-Trends.md
4. 提取 6 个关键概念：
   - Agentic AI → 创建 4-Resources/tech/ai/Agentic-AI.md
   - 多模态融合 → 创建 4-Resources/tech/ai/多模态融合.md
   - ... (其余 4 篇)
5. 搜索 vault，发现已有 [[AI智能体]] → 链接到新笔记
6. 更新 MOC-人工智能.md，添加新概念引用
7. 输出消化报告：1 篇来源 + 6 篇概念笔记 + 摘要
```

### Skill 4：vault-auditor（知识库审计官）

**标识符**：`vault-auditor`

**定位**：定期审计知识库的健康状况，发现需要关注的问题并生成可操作的改进建议。

#### 核心能力

| 能力 | 描述 |
|------|------|
| 质量评分 | 基于完整性、链接数、更新频率为笔记打分 |
| 陈旧检测 | 标记超过指定时间未更新的笔记 |
| 存根识别 | 找出内容过短、仅占位的"存根笔记" |
| 重复检测 | 识别标题或内容高度相似的笔记对 |
| 一致性检查 | 检查属性使用、命名规范、标签一致性 |
| 归档建议 | 标记已完成但未归档的项目 |
| 统计概览 | 生成 vault 整体统计报告 |

#### 输入定义

```
Input:
  scope:
    mode: "global" | "directory" | "tag"
    value: string?                    // directory 时指定路径，tag 时指定标签
  checks:
    - "quality"       // 质量评分
    - "stale"         // 陈旧检测
    - "stub"          // 存根检测
    - "duplicate"     // 重复检测
    - "consistency"   // 一致性检查
    - "archive"       // 归档建议
    - "all"           // 全部检查
  thresholds:
    stale_days: number (default: 180)
    stub_min_chars: number (default: 200)
    duplicate_similarity: number (default: 0.7)
```

```
Output:
  report:
    generated: datetime
    scope: string
    overall_score: "A" | "B" | "C" | "D" // 综合健康度
    stats:
      total_notes: number
      total_size: string
      notes_by_type: { [type]: number }
      avg_quality_score: number
    issues:
      critical: Issue[]                 // 必须处理
      warning: Issue[]                  // 建议处理
      info: Issue[]                     // 仅供参考
    suggestions: Suggestion[]           // 具体改进建议
```

#### 质量评分维度

| 维度 | 权重 | 评估标准 |
|------|------|----------|
| 内容完整性 | 30% | 正文长度、结构完整性（有标题/正文/链接） |
| 链接丰富度 | 25% | 出链数 + 入链数 |
| 属性完整性 | 20% | 必填属性是否齐全 |
| 更新时效 | 15% | 最近更新距今时间 |
| 格式规范 | 10% | 是否符合命名和格式约定 |

#### 边界约束

- **完全只读**：不修改任何文件。
- **评分仅供参考**：不做自动化删除或合并决策。
- **重复检测使用组合特征**：标题相似度 + 标签重叠 + 内容关键词重叠，避免误判。
- **隐私保护**：审计报告不包含笔记正文全文，仅包含路径和统计信息。
- **排除项可配置**：特定目录/文件可标记为排除审计。

#### 触发条件

- "审计一下知识库"
- "检查 vault 健康度"
- "哪些笔记需要更新了？"
- "找出重复笔记"
- "检查笔记质量"
- 建议配置为定期自动任务（每月一次）

#### 使用示例

```
用户: "全面审计一下知识库"

Agent 行为:
1. 扫描所有非系统目录的 .md 文件
2. 对每个文件计算质量评分
3. 检查陈旧（>180天未更新）→ 找到 12 篇
4. 检查存根（<200字符）→ 找到 5 篇
5. 检查重复 → 找到 3 组疑似重复
6. 检查一致性 → 发现 8 篇缺少 status 属性
7. 检查归档 → 发现 2 个已完成项目未归档
8. 生成审计报告，按严重程度排序
9. 给出综合评分 + 改进建议 Top 10
```

### Skill 5：source-curator（来源策展人）

**标识符**：`source-curator`

| 来源笔记 | 为标准化的来源元笔记（书/文章/视频/课程等） |

| 知识整合 | 将来源中提取的概念链接到已有知识网络 |
| 推荐发现 | 基于已有来源推荐相关阅读 |

#### 输入定义

```

  action: "add" | "update" | "summarize" | "list" | "integrate"
  source:
    type: "book" | "article" | "paper" | "video" | "podcast" | "course"
    title: string
    author: string?
    url: string?
    notes: string?                    // 用户附注
  progress:
    status: "want-to-read" | "reading" | "completed" | "abandoned"

Output:
  source_path: string                 // 来源笔记路径
  reading_list_updated: boolean
  summary: string?                    // summarize 动作时返回

#### 工作流

**添加来源**：
```
接收来源信息 → 验证必填字段 → 搜索 vault 避免重复 → 使用 tpl-source 模板创建来源笔记 → 更新待读清单

**更新进度**：
```

**整合知识**：
```
定位已完成的来源笔记 → 提取关键概念 → 搜索 vault 中概念 → 建立双向链接 → 提示是否需要创建新概念笔记 → 建议更新相关 MOC

- **个人评注与原文区分**：使用 callout 明确区分 `> [!quote]`（原文引用）和 `> [!thought]`（个人思考）。
- **元数据容错**：部分来源元数据允许缺失，但标题和类型为必填。
- **去重逻辑**：同标题+同作者的来源视为重复，提示用户确认。
- **弃读也记录**：abandoned 状态的来源也需要记录弃读原因，供未来参考。

#### 触发条件

- "添加到待读列表..."

### Skill 6：journal-enhancer（日记增强器）

**标识符**：`journal-enhancer`

**定位**：提升日常记录的效率，建立从日记到永久笔记的提炼管道，让日记不仅是流水账，更是知识原材料。

| 知识提炼 | 识别日记中的可转化为永久笔记的内容 |
| 周期回顾 | 生成周/月/季度回顾摘要 |
| 习惯追踪 | 从日记中提取习惯数据，生成趋势 |

```
Input:
  action: "create" | "review" | "extract" | "reflect"

    type: "daily" | "weekly" | "monthly" | "quarterly"

用户: "回顾这个月，提取知识点"

Agent 行为:
1. 读取 5-Journal/daily/2026/05/*.md 所有日记
2. 分析内容，识别：
   - 反复出现的主题 → 可能是一个需要创建的领域笔记
   - 明确的"学到了..."、"发现..."表述 → 候选知识笔记
   - 项目完成后的反思 → 可提炼为项目复盘笔记
3. 建议：
   - "关于 Obsidian 模板系统的经验" → 建议创建 4-Resources/tools/obsidian-templates.md
   - "项目管理中的沟通技巧反思" → 建议与已有 [[项目沟通]] 合并
4. 用户确认后，创建/更新相应笔记
```

## 五、工作流体系

### 5.1 日常流程（每日 10 分钟）

```
早晨（3 分钟）
  创建今日日记 → 回顾昨日 → 确认今日任务
  触发技能：journal-enhancer (create daily)

白天（随时）
  有想法 → 扔进 0-Inbox
  读文章 → info-digester 消化 → 笔记进入 7-Sources + 4-Resources
  写笔记 → note-composer 辅助 → 确保格式和链接规范

晚间（7 分钟）
  处理 0-Inbox → 清空收件箱
  完善今日日记 → 反思
  触发技能：journal-enhancer (extract) → 从日记提炼知识
```

### 5.2 每周流程（每周日 30 分钟）

```
1. 创建本周回顾（journal-enhancer weekly review）
2. 审计本周新增笔记质量（vault-auditor targeted）
3. 检查并修复孤岛笔记（knowledge-linker orphans）
4. 更新相关 MOC（knowledge-linker moc-gap）
5. 规划下周项目
```

### 5.3 每月流程（每月最后一天 1 小时）

```
1. 创建本月回顾（journal-enhancer monthly review）
2. 全面知识库审计（vault-auditor global）
3. 处理审计发现的问题
4. 待读清单整理（source-curator list）
5. 归档已完成项目
6. 更新知识库设计规范（如有需要）
```

### 5.4 摄入工作流

这是知识从外部进入 vault 的标准路径：

```
发现内容 → source-curator (add to reading list)
              ↓
         阅读/学习
              ↓
         读完后 → source-curator (mark completed)
              ↓
         info-digester (digest content)
              ↓
    ┌────────┼────────┐
    ↓        ↓        ↓
来源笔记   概念笔记   MOC 更新
(7-Sources)(4-Resources)(1-Atlas)
              ↓
         knowledge-linker (link to existing)
              ↓
         知识网络更新完成
```

## 六、技能实现优先级

| 优先级 | 技能 | 理由 |
|--------|------|------|
| P0 | note-composer | 最基础：日常写笔记的第一步 |
| P0 | info-digester | 最刚需：外部信息进入 vault 的主入口 |
| P1 | knowledge-linker | 释放 Obsidian 图数据库的核心价值 |
| P1 | source-curator | 与 info-digester 配合，形成完整摄入链路 |
| P2 | vault-auditor | 需要 vault 有一定规模后才发挥作用 |
| P2 | journal-enhancer | 依赖日记积累，初期可手动替代 |

## 七、技术依赖

### 各技能依赖的 SDK / 工具

```
note-composer:
  └── obsidian-cli (搜索), obsidian-markdown (生成)

info-digester:
  └── defuddle (网页), pdf-skill (PDF), obsidian-markdown (生成),
      obsidian-cli (搜索已有笔记)

knowledge-linker:
  └── obsidian-cli (搜索、反向链接、文件列表)

vault-auditor:
  └── obsidian-cli (文件列表、元数据读取)

source-curator:
  └── obsidian-cli (读写), obsidian-markdown (模板)

journal-enhancer:
  └── obsidian-cli (搜索、读写), obsidian-markdown (模板)

### 已安装的可用技能

- `obsidian-cli` — Obsidian vault 的 CLI 操作
- `obsidian-markdown` — Obsidian 风格的 Markdown 生成
- `json-canvas` — JSON Canvas 文件（可用于可视化知识图谱）
- `obsidian-bases` — Obsidian Bases（可用于数据库视图）
- `defuddle` — 网页内容提取
- `pdf` — PDF 处理
